# PDF 工具设计文档

日期：2026-03-24
项目：`pdf-split`

## 1. 目标

在当前 Tauri 应用中实现两个桌面端 PDF 工具：

1. 选择一个 PDF 文件，将 PDF 按页导出为图片。
2. 选择一个 PDF 文件，生成带文字水印的新 PDF。

当前应用负责界面、文件选择、参数收集、调用 Rust 命令以及展示结果。

实际的 PDF 处理能力由本地依赖 `kx-pdf` 提供。

## 2. 已确认需求

### 2.1 PDF 转图片

- 用户选择一个 PDF 文件。
- 用户选择一个输出目录。
- 用户可以选择图片格式：`PNG` 或 `JPG`。
- 程序按页导出图片，每页一张。

### 2.2 文字水印

- 用户选择一个 PDF 文件。
- 用户选择一个输出目录。
- 用户在页面输入水印文字。
- 程序生成一个新的带水印 PDF 文件。

## 3. 职责边界

### 3.1 `pdf-split` Tauri 应用负责

- 渲染单页工具界面。
- 选择 PDF 文件。
- 选择输出目录。
- 收集用户参数。
- 调用 Tauri 命令。
- 展示加载中、成功、失败状态。

### 3.2 `pdf-split` Tauri 应用不负责

- 将 PDF 页面渲染成图片。
- 直接修改 PDF 页面内容。
- 实现水印绘制逻辑。

### 3.3 本地依赖 `kx-pdf` 负责

- 将 PDF 按页渲染为图片。
- 按 `PNG` 或 `JPG` 输出图片文件。
- 为 PDF 页面添加文字水印。
- 写出最终生成的 PDF 文件。
- 处理底层 PDF 解析、绘制与文件输出细节。

## 4. 当前技术上下文

- 前端：React 19 + Vite
- 桌面容器：Tauri 2
- Rust 应用目录：`src-tauri`
- 当前本地 PDF 依赖：

```toml
kx-pdf = { version = "0.1", path = "/Volumes/data/code/rust/kx/crates/pdf" }
```

当前 `kx-pdf` 已有部分 PDF 相关能力，但本项目所需的“按页转图片”和“文字水印”能力，应补充在本地 `kx-pdf` 中，再由当前项目进行调用。

## 5. 界面设计

应用采用单页结构，页面中包含两个互相独立的功能卡片。

### 5.1 卡片 A：PDF 转图片

字段：

- PDF 文件路径
- 输出目录路径
- 图片格式选择：`PNG` / `JPG`

操作：

- 选择 PDF
- 选择输出目录
- 开始导出

结果展示：

- PDF 总页数
- 输出目录
- 实际生成的图片数量

### 5.2 卡片 B：文字水印

字段：

- PDF 文件路径
- 输出目录路径
- 水印文字

操作：

- 选择 PDF
- 选择输出目录
- 开始生成水印 PDF

结果展示：

- 输出 PDF 路径
- 完成状态

### 5.3 交互规则

- 两个卡片拥有各自独立的表单状态。
- 两个卡片拥有各自独立的处理中状态。
- 任务执行时，对应按钮禁用。
- 第一版只展示“处理中 / 成功 / 失败”状态，不做逐页进度条。

## 6. 输出规则

### 6.1 图片导出命名

建议文件名格式：

- `原文件名-p001.png`
- `原文件名-p002.png`
- `原文件名-p001.jpg`

设计原因：

- 文件名按字典序排序稳定。
- 页面编号清晰。
- 避免 `1、10、2` 这样的排序问题。

### 6.2 水印 PDF 命名

建议文件名格式：

- `原文件名-watermarked.pdf`

规则：

- 第一版不覆盖原始 PDF 文件。

### 6.3 同名文件冲突处理

第一版行为：

- 如果目标输出文件已经存在，直接返回明确错误，提示用户更换目录或删除已有文件。

设计原因：

- 防止误覆盖已有文件。
- 第一版行为更简单、可预期。

## 7. 水印样式规则

第一版的水印样式固定在 PDF 库实现中，不在页面上开放样式参数。

建议默认样式：

- 文字水印
- 浅灰色
- 斜向旋转
- 在页面中重复铺设
- 带一定透明效果

第一版页面只允许用户输入水印文字，不允许配置字号、颜色、透明度、角度等参数。

## 8. 命令接口设计

Tauri 层对前端暴露两个命令。

### 8.1 `split_pdf_to_images`

输入：

- `input_path: String`
- `output_dir: String`
- `image_format: String`

输出：

- 结构化结果，至少包含：
  - 页数
  - 生成文件数量
  - 输出目录

### 8.2 `add_text_watermark`

输入：

- `input_path: String`
- `output_dir: String`
- `watermark_text: String`

输出：

- 结构化结果，至少包含：
  - 输出 PDF 路径

Tauri 层命令应尽量保持轻量，只做参数校验、调用 `kx-pdf`、结果转换。

## 9. 校验规则

### 9.1 前端校验

以下情况直接阻止调用命令：

- 未选择 PDF 文件
- 未选择输出目录
- 水印功能未输入文字
- 图片导出功能未选择格式

### 9.2 Rust 命令层校验

以下情况返回明确错误：

- 输入文件不存在
- 输入文件不是合法 PDF
- 输出目录不可写
- 水印文字为空
- PDF 处理失败

## 10. 错误处理

前端只展示用户可理解的错误文案。

Rust 命令层负责：

- 做输入校验
- 调用 `kx-pdf`
- 将内部处理错误映射为稳定、可展示的错误信息

前端不直接展示 panic、堆栈信息或底层实现细节。

## 11. 测试策略

### 11.1 当前 Tauri 项目测试重点

- 参数校验
- 输出命名规则
- 命令返回结果映射
- 前端成功与失败状态展示

### 11.2 `kx-pdf` 测试重点

- PDF 按页渲染图片
- `PNG` / `JPG` 输出正确性
- 带文字水印 PDF 生成
- 不覆盖已有文件
- 非法 PDF 输入处理

由于当前项目不实现底层 PDF 算法，深度 PDF 产物校验应主要放在 `kx-pdf` 中完成。

## 12. 实现说明

- 当前项目后端代码应尽量保持轻量，只做编排与调用。
- 核心 PDF 能力统一沉淀在本地依赖 `kx-pdf` 中，不在当前项目重复实现。
- 前端页面应替换默认的 Tauri 模板，改为面向生产使用的 PDF 工具页。

## 13. 验收标准

满足以下条件即可认为本功能完成：

1. 页面可以分别为两个功能选择 PDF 文件和输出目录。
2. 图片导出功能可以按所选格式成功输出每一页图片。
3. 文字水印功能可以根据输入文字生成新的带水印 PDF。
4. 应用能清晰展示成功与失败状态。
5. 原始 PDF 文件不会被覆盖。
6. 当前 Tauri 项目只负责编排与调用，真实 PDF 处理能力由 `kx-pdf` 提供。
